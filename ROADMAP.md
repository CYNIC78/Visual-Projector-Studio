# ROADMAP — Prompt Node v2 и Asset Gateway

Чего мы хотим добиться: чтобы пользователь мог взять картинку из галереи, кинуть её в ноду,
написать пачку промптов, нажать одну кнопку — и получить кучу готовых ассетов,
которые уже лежат в галерее и ждут своего часа.

Без шаманства, без тонны ручной работы, без копания в файлах.

---

## Фаза 1 — Фундамент Prompt Node

**Цель:** переделать ноду промпта так, чтобы в ней можно было хранить много вариантов,
а Produce All прогонял всё сразу.

### Что меняем

**1. Внутренняя структура ноды**

Раньше:
```
this.data.positive = "текст"
this.data.negative = ""
this.data.reference = ["путь1", "путь2"]
```

Стало:
```
this.data.reference = ["путь1", "путь2"]
this.data.tabs = [
  { id: "tab_abc", name: "smile", text: "Make character smile..." },
  { id: "tab_def", name: "angry", text: "Make character angry..." },
]
this.data.activeTabId = "tab_abc"
```

Самый важный момент: **текст таба — это то, что пишет пользователь.**
Никакой магии — просто текст промпта для этой конкретной генерации.

**2. Команды**

В тексте можно писать `{name:emily_smile}` — это наша команда.
Перед отправкой в модель фигурные скобки вместе с содержимым вырезаются.
На основе `{name:...}` определяется тег итогового ассета в галерее.

Пока команда одна. В будущем появятся другие. Синтаксис заложен — `{...}` вырезается всегда.

**3. Референсы**

Общие на всю ноду. Сейчас там дропзона, работает. Чуть позже докрутим интеграцию с галереей.

**4. Кнопки**

- Активный таб → «Generate» или «Produce Active» — генерирует один ассет.
- «Produce All» — перебирает все табы по очереди, каждый даёт свой ассет.
- Стоп, прогресс-бар — уже есть в студии.

**5. Именование ассетов**

Приоритет:
1. `{name:вариант}` из текста → тег `вариант`
2. Нет команды → тег `gen_{счетчик}` (просто техническое имя, без смысла)

Коллизии решает `getUniqueImportedTag()` из галереи — добавляет `_1`, `_2`.

### Не делаем в этой фазе

- Многоосевые табы, декартово произведение, `{tab:...}` ссылки.
- Drag-n-drop из галереи в референсы (будет в Фазе 2).
- Категории и группировку табов в ноде (будет, если табов станет реально много).

## Фаза 2 — Интеграция с галереей

**Цель:** чтобы картинки из галереи можно было кидать в референсы ноды,
а готовые ассеты летели обратно в галерею и сразу показывались.

### Референсы из галереи

Сейчас дропзона в промпт-ноде ловит `dataTransfer.files` (файловые пути).
А drag-n-drop из галереи кидает `text/plain` с `[IMG:tag]` и `vp/asset-move-batch`.

Надо:
- Научить дропзону распознавать `vp/asset-move-batch`.
- Доставать ассеты из `S.gallery` по тегу.
- Брать их blob / base64 / url и сохранять в `this.data.reference`.

### Отображение галерейных ассетов в ноде

Когда в референсы попадает ассет из галереи — показывать его превью как обычно.
Никакой разницы с файловыми референсами с точки зрения UI.

### Производство сразу в галерею

`displayResult()` уже вызывает `addImageFromBlob()`. Там всё нормально,
но надо убедиться что `suggestedName` корректно формируется из команды `{name:...}`.

## Фаза 3 — Продвинутые штуки (когда будет нужно)

### Многоосевые табы (декартово произведение)

Появится команда `{tab:poses}` — ссылка на другой таб внутри ноды.
Система будет перебирать все комбинации и генерировать каждую.

### Data Tab

Таб может быть не промптом, а набором строк (вариантов).
Активные промпт-табы ссылаются на него через `{tab:...}`.

### Категории табов

Когда табов сотни — сгруппировать их по папкам прямо внутри ноды.
Как в галерее сейчас: категории > табы > ассеты.
Только здесь: категории > табы > промпты.

---

## Принципы (не нарушать)

1. **Пользователь не должен думать о тегах, путях и файлах.** Всё должно работать через UI.
2. **Одна кнопка — одна пачка.** Produce All делает всё сам.
3. **Никакого оверинжиниринга.** Если фичу можно не делать — не делаем.
4. **Совместимость.** Старые воркфлоу (которые `this.data.positive`) не ломаются.
5. **Текст — главный.** Модель ест текст. Всё, что в `{...}`, — служебное, не для модели.

---

## Симметрия: модель = пользователь

Проект спроектирован симметрично: всё, что может делать пользователь через UI,
должно быть доступно модели через команды. Это касается и Asset Studio.

**Модель должна уметь:**

1. Создавать табы в Prompt Node.
2. Писать в них промпты.
3. Заполнять референсы нужными ассетами.
4. Запускать Produce All.
5. Применять результаты.

**Как это выглядит на практике:**

Пользователь пишет:
> «Гемма, а давай на основе emily.png создадим пачку эмоций: улыбка, грусть, злость, удивление.»

Модель генерирует **код настройки Prompt Node**, который пользователь применяет
(одним кликом или автоматически). Код:

- Создаёт табы: `smile`, `sad`, `angry`, `surprised`
- Пишет в каждый промпт: `{name:emily_smile} Make character smile...`
- Ставит `emily.png` в референсы
- Запускает Produce All

**Механизм:**

Где-то в системе живёт экшн (пока не реализован, но закладываем),
который модель вызывает как тулзу. Экшн принимает JSON-описание конфигурации ноды:

```json
{
  "action": "configure_prompt_node",
  "referenceTags": ["emily"],
  "tabs": [
    { "name": "smile", "text": "{name:emily_smile} Make the character smile warmly...\nKeep all features, lighting and background intact." },
    { "name": "sad",   "text": "{name:emily_sad} Make the character look sad...\nTearful eyes, downturned lips..." },
    { "name": "angry", "text": "{name:emily_angry} Make the character angry...\nFurrowed brows, clenched jaw..." }
  ]
}
```

Пользователь видит результат, проверяет, применяет. Дальше — Produce All,
и пачка ассетов уже в галерее.

**Что это даёт:**

- Модель берёт на себя рутину: придумать вариации, написать промпты, разложить по табам.
- Пользователь только контролирует качество и нажимает «применить».
- Полная симметрия: что умеет пользователь в UI — умеет и модель через команды.
